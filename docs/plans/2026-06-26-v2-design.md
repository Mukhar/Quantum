# Quantum v2 — Design Doc

*Created: 2026-06-26 via brainstorming session with huppy.*
*Status: Design — not yet planned into `.planning/ROADMAP.md` phases.*
*v1 context: v1 launches at the end of Phase 5 (algorithm track). v2 starts after.*

---

## 1. Problem Statement

### In plain language

After v1 ships, the Quantum site is a polished read-and-play experience —
seven essays, one sandbox with URL-fragment sharing, generative-art and
audio outputs. But three "next-step" itches will be obvious by then:

1. Readers who build cool circuits in the sandbox have no way to **save**
   them locally. They can copy URLs around, but there's no "my circuits"
   shelf to come back to.
2. The site ships **one theme** (per v1 decision). Readers who live in
   dark mode at night get blasted. Accessibility + comfort hit.
3. We have **no channel** for readers to send feedback, suggest new
   essay topics, or report bugs without leaving the site for GitHub.

v2 closes those three gaps **without breaking the static-site, no-backend
posture that defines the project.**

### Who is affected

- **Returning readers** — they accumulate "favorite circuits" in their
  head with nowhere to put them
- **Night-mode readers** — currently squinting
- **Power users / educators** — want to send "hey, please cover topic X"
  without finding the GitHub repo

### What exists today

- Sandbox URL-fragment sharing (Phase 3 of v1) covers *transient* sharing
  but not *personal persistence*. Closing the tab loses the circuit
  unless you saved the URL somewhere external.
- No theme system. Single hardcoded palette.
- No in-site contact channel. Implicit: "open a GitHub issue."

## 2. Solution Overview

### In plain language

Three additions to v1, all client-side or via free third-party form
handlers — **no backend, no servers, no ops cost.**

1. **Circuit Gallery** — a "my saved circuits" shelf in the browser,
   backed by IndexedDB. Save, name, load, delete, duplicate, export,
   import. Data structure designed so it can sync to a remote API later
   with zero migration.
2. **Dark + Light Mode** — Tailwind `darkMode: 'class'` theme system,
   stored in `localStorage`, defaults to OS preference. Inline `<head>`
   script kills FOUC. Visual audit pass across all widgets.
3. **Unified Feedback Form** — one `/feedback` page, one form with a
   Type dropdown (general / topic suggestion / feature request / bug).
   POSTs to a Google Apps Script Web App that appends rows to a private
   Google Sheet you own. Honeypot for spam.

### Visual map of the additions

```
                       ┌──────────────────────────────┐
                       │       v1 site (existing)     │
                       │  essays + /sandbox + URLs    │
                       └──────────────────────────────┘
                                      │
            ┌─────────────────────────┼─────────────────────────┐
            │                         │                         │
            ▼                         ▼                         ▼
   ┌────────────────┐        ┌────────────────┐        ┌────────────────┐
   │ Theme system   │        │ Gallery        │        │ /feedback page │
   │ (every page)   │        │ (sandbox+page) │        │ (one route)    │
   └────────────────┘        └────────────────┘        └────────────────┘
            │                         │                         │
            ▼                         ▼                         ▼
   localStorage["theme"]       IndexedDB["quantum/         Google Apps Script
   + <html class="dark">        gallery"]                  → Google Sheet
                                                            (owner-only)
```

### Key design principle (carried from v1)

**"No backend"** means no servers *you* run. Third-party HTTP endpoints
(Google Apps Script, Formspree, etc.) are acceptable because they cost
nothing, require no ops, and can be replaced with a real API later
without breaking the client contract.

## 3. Architecture & Components

### 3.1 Circuit Gallery

**Storage:** IndexedDB via [`idb-keyval`](https://github.com/jakearchibald/idb-keyval) (~600 bytes gzipped).
Chosen over `localStorage` because:
- `localStorage` caps at 5 MB and stringifies everything synchronously
- IndexedDB scales to hundreds of MB and is async (no main-thread jank
  when saving a circuit with a thumbnail data URL)
- `idb-keyval` exposes a `get/set/del/keys` API that matches our mental
  model (one key per circuit by ID)

**Entry shape** (locked v1 of the gallery schema — bump `schemaVersion`
to migrate later):

```ts
interface GalleryEntry {
  id: string;              // ULID — sortable + unique, ~50 bytes
  schemaVersion: 1;        // bumped on shape changes; reader migrates
  name: string;            // user-supplied; default "Untitled circuit"
  createdAt: number;       // ms since epoch
  updatedAt: number;       // ms since epoch
  qubits: number;          // 1..4
  steps: number;           // 1..~20
  circuit: CircuitData;    // same shape as sandbox URL-fragment codec
  thumbnailDataUrl?: string; // small PNG, ≤ 8 KB; generated on save
  tags?: string[];         // optional, future-friendly
}
```

**Why this shape maps cleanly onto a future API:** a REST endpoint
`POST /circuits` would accept *literally this JSON*. The sync layer
becomes "for each local entry without a remoteId, POST it; for each
remote entry not locally present, save it." No re-architecture.

**Files added:**

```
src/lib/gallery/
  store.ts          ← idb-keyval wrapper, CRUD operations
  thumbnail.ts      ← renders circuit → 200×80 PNG dataURL
  schema.ts         ← types + migrations
  store.test.ts     ← vitest with fake-indexeddb
src/components/sandbox/
  GalleryDrawer.tsx ← slide-out panel from /sandbox (save/load UI)
src/pages/
  gallery.astro     ← dedicated /gallery browse page
```

### 3.2 Dark + Light Mode

**Mechanism:** Tailwind's `darkMode: 'class'` strategy. A `dark` class on
`<html>` flips every `dark:bg-…` / `dark:text-…` utility.

**State:** `localStorage["quantum/theme"]` = `"light" | "dark" | "system"`.
Default is `"system"` (follows OS `prefers-color-scheme`).

**FOUC prevention:** Inline blocking script in `<head>` of `BaseLayout.astro`,
runs *before* any CSS is applied:

```html
<script is:inline>
  (function () {
    const stored = localStorage.getItem('quantum/theme');
    const prefersDark = window.matchMedia('(prefers-color-scheme: dark)').matches;
    const useDark = stored === 'dark' || (stored !== 'light' && prefersDark);
    if (useDark) document.documentElement.classList.add('dark');
  })();
</script>
```

**Toggle UI:** A 3-state button in the site header (Light / Dark / System).
Persists immediately, applies class to `<html>`.

**Widget audit checklist** (every existing widget must be touched):
- [ ] KaTeX math blocks — explicit `dark:text-slate-100`, override
      KaTeX's own white-background assumptions
- [ ] Three.js Bloch sphere — scene background tied to a CSS-var color;
      arrow/axis colors need dark variants
- [ ] ProbabilityBars — bar colors must hold contrast on both themes
- [ ] StateVector readout — code-font color
- [ ] Sandbox grid + gate palette
- [ ] Quantum Canvas — palette generation must produce visible colors
      on either background (or paint its own background)
- [ ] Quantum Tones — keyboard visualization
- [ ] Annotations / pinned notes
- [ ] Concept-map homepage (Phase 4 of v1)
- [ ] Code blocks (Astro Shiki or similar)

**No architectural alternatives worth listing.** Tailwind class-based
dark mode is the boring-correct answer for an Astro+Tailwind site.

### 3.3 Unified Feedback Form

**Page:** `/feedback` — a single Astro page with a vanilla HTML form
(no JS framework needed for one form).

**Form fields:**

| Field | Type | Notes |
|---|---|---|
| Type | `<select>` | "General feedback" / "Topic suggestion" / "Feature request" / "Bug report" |
| Subject | `<input type="text">` | Required, max 100 chars |
| Message | `<textarea>` | Required, max 2000 chars |
| Email | `<input type="email">` | Optional ("if you want a reply") |
| Page context | hidden `<input>` | Auto-filled with `document.referrer` |
| `_hp` honeypot | hidden `<input>` | If non-empty on submit, drop server-side |

**Transport:** HTML form `action` POSTs to a **Google Apps Script Web App
URL** you deploy. The Apps Script (~15 lines) appends a row to a Google
Sheet you own. Response is a 302 redirect to `/feedback/thanks`.

**Why Apps Script over Formspree/Web3Forms:**

| | Apps Script + Sheet | Formspree free tier |
|---|---|---|
| Cost | $0 forever | $0 up to 50 submissions/month |
| Data ownership | Your Google account, your sheet | Their dashboard, their servers |
| Submission cap | None practical | 50/mo, then paid |
| Setup time | ~10 min (one-time copy-paste) | ~2 min |
| Spam handling | Honeypot in client + Apps Script filter | Their built-in filter |

You only pay the ~8 extra setup minutes once. Worth it.

**Files added:**

```
src/pages/feedback/
  index.astro      ← the form
  thanks.astro     ← post-submit confirmation
docs/
  apps-script.md   ← copy-paste setup guide for the Apps Script Web App
```

## 4. Data Flow

### 4.1 Save a circuit from the sandbox

```
User in /sandbox
   │
   ├──► Clicks "Save to gallery" in toolbar
   │
   ├──► GalleryDrawer slides out, prompts for name
   │
   ├──► thumbnail.ts renders circuit → 200×80 PNG dataURL (<8KB)
   │
   ├──► store.ts builds GalleryEntry { id: ulid(), schemaVersion: 1, ... }
   │
   ├──► idb-keyval.set(`circuit:${id}`, entry)
   │
   └──► Toast: "Saved as '<name>' • View gallery"
```

### 4.2 Load a circuit from the gallery page

```
User opens /gallery
   │
   ├──► On mount: idb-keyval.keys() → filter "circuit:*" → batch get()
   │
   ├──► Render grid of cards (thumbnail + name + updatedAt)
   │
   ├──► User clicks card
   │
   ├──► Navigate to /sandbox#<urlEncodedCircuit>  (existing v1 codec)
   │
   └──► Sandbox hydrates from URL fragment — no special "load" code path
```

**Pedantic DRY note:** Loading a gallery entry **reuses the existing
URL-fragment hydration path.** We don't write a second "load circuit
into the sandbox" code path. The gallery is just a place where URL
fragments live; clicking one is identical to pasting one.

### 4.3 Submit feedback

```
User on /feedback
   │
   ├──► Fills form, clicks Submit
   │
   ├──► Browser POSTs form-encoded body to Apps Script Web App URL
   │
   ├──► Apps Script: validates honeypot empty, appends row to Sheet
   │
   ├──► Returns 302 → /feedback/thanks
   │
   └──► Thanks page shows confirmation + "send another" link
```

### 4.4 Theme toggle

```
Page load
   │
   ├──► Inline <head> script reads localStorage["quantum/theme"]
   │     and prefers-color-scheme → sets <html class="dark"> if needed
   │
   ├──► CSS applies → no FOUC
   │
   ├──► Later, user clicks theme toggle
   │
   ├──► Writes localStorage["quantum/theme"] = newValue
   │
   └──► Toggles <html> class immediately
```

## 6. Error Handling & Edge Cases

### Gallery

| Case | Behavior |
|---|---|
| IndexedDB unavailable (Safari private browsing, etc.) | Catch, show banner: "Your browser is blocking local storage; circuits won't persist." Fall back to in-memory list for the session. |
| Storage quota exceeded | Toast: "Storage full — delete some circuits or export to file." |
| Schema version mismatch on read | `schema.ts` migrates known versions; unknown versions are quarantined into a "needs migration" bucket, not deleted. |
| Thumbnail render fails | Save the entry without `thumbnailDataUrl`; gallery card shows a generic icon. |
| Import a malformed JSON file | Validate against schema, show inline errors, don't blow away existing entries. |

### Theme

| Case | Behavior |
|---|---|
| `localStorage` blocked (privacy mode) | Inline script try/catches; defaults to `prefers-color-scheme`. Toggle still works for the session. |
| Three.js scene live during theme toggle | Scene's `background` is set from CSS var, re-read on toggle event. |

### Feedback form

| Case | Behavior |
|---|---|
| Honeypot field non-empty (bot) | Apps Script drops silently with 200 OK (no signal to the bot). |
| Apps Script endpoint down / network error | Form falls back to `mailto:` link with prefilled subject/body. |
| Submission > 2000 chars | Client-side `maxLength` blocks; server-side truncate as belt + braces. |
| User submits 10× in a minute | Apps Script throttles by hashed IP (Sheet rate-limit column). Soft cap: 5/min. |

## 7. Testing Strategy

### Gallery
- `src/lib/gallery/store.test.ts` — CRUD operations against
  [`fake-indexeddb`](https://www.npmjs.com/package/fake-indexeddb)
- Schema migration test (write a v0 entry manually, read with v1
  reader, assert correct migration)
- Round-trip test: save circuit → load from gallery → verify circuit
  state equals original (re-uses sandbox codec tests)

### Theme
- Vitest component test: toggle button cycles through 3 states,
  writes to localStorage, mutates `<html>` class
- Playwright visual diff: snapshot every page in both themes; fail
  on contrast regression below WCAG AA

### Feedback form
- Manual: deploy the Apps Script, submit one of each Type, verify
  rows appear in the Sheet with correct columns
- Vitest: form validation (required fields, max lengths, honeypot
  field is hidden from real users via `tabindex=-1` and `aria-hidden`)

### Cross-cutting
- Lighthouse a11y ≥ 95 on `/gallery` and `/feedback`, both themes
- Bundle size: gallery code lazy-loaded from `/sandbox` and `/gallery`
  only; doesn't bloat essay pages

## 8. Open Questions & Risks

### Open questions (decide before plan phase)

1. **Cloud sync trigger:** When (and *whether*) to add API-backed sync.
   Not in v2 scope per user direction, but the gallery schema is built
   to accommodate it. Revisit at v2 retrospective.
2. **Gallery sharing UX:** Should each gallery entry expose a "copy
   shareable URL" button? (Probably yes — it's trivial: stringify the
   circuit through the existing URL codec. Punt to plan phase.)
3. **Anonymous vs identified feedback:** The form has an optional email
   field, but should we capture a "session ID" (random UUID in
   localStorage) so repeat feedback from the same person is linkable
   in the sheet? Lightweight, no PII. Probably yes.
4. **Theme toggle in essays' scroll-tied animations:** Phase 2 essays
   use scroll-driven transforms; verify they don't depend on a fixed
   theme assumption.

### Risks (from reverse brainstorming)

| Risk | Severity | Mitigation |
|---|---|---|
| FOUC on first paint | High (visible) | Inline `<head>` script Day 1 |
| Spam floods the sheet | Medium | Honeypot + Apps Script rate-limit + (optional) simple math captcha |
| Three.js / KaTeX dark-mode visual bugs | Medium | Mandatory widget audit checklist (§3.2) before v2 ship |
| Gallery schema drift breaks saved entries | Medium | `schemaVersion` + migration tests Day 1 |
| IndexedDB hostile in private browsing | Low | Try/catch + in-memory fallback + warning banner |
| Apps Script URL exposed in client HTML | Low | It's *supposed* to be public; rate-limit defends against abuse |
| Users save 500 circuits, thumbnails balloon storage | Low | Soft warning at 100 entries; "manage storage" link |

## 9. Suggested Sequencing (4 sub-phases)

Each phase independently shippable. Smallest-to-largest order picked
deliberately so theme system is in place *before* gallery UI is built
on top (avoiding double-audit).

| Phase | Scope | Why this order |
|---|---|---|
| **v2-01** | Dark + light mode (toggle, FOUC script, full widget audit) | Touches every page. Ship first so subsequent v2 work is theme-aware from birth. |
| **v2-02** | Unified feedback form (`/feedback` + Apps Script + Sheet) | Tiny, self-contained, gives you a feedback channel *during* the rest of v2 dev. |
| **v2-03** | Circuit gallery (IndexedDB store, `/gallery` page, sandbox drawer) | Meatiest feature. Depends on existing sandbox circuit codec. |
| **v2-04** | v2 polish + launch (gallery export/import, a11y/Lighthouse audit, dark-mode visual QA, v2 announcement) | Standard closing-the-loop work. |

## 10. Out of Scope (still v3+ territory)

- Multi-user collaborative editing of circuits
- Public circuit gallery / community feed
- Comment threads (replaced by feedback form in v2)
- Upvoting / Canny-style public roadmap
- User accounts / login
- Sync gallery across devices (schema is *ready* but the sync layer is v3)
- Real quantum hardware integration
- i18n
- Mobile-native app

## 11. Decisions Log

| Decision | Rationale | Alternative considered |
|---|---|---|
| IndexedDB via `idb-keyval` over localStorage | 5MB cap too tight; async API; future-friendly | Plain localStorage |
| Merge feedback + feature-request forms into one | DRY; same shape, just a category column | Separate `/feedback` and `/suggest` pages |
| Google Apps Script + Sheet over Formspree | Free forever, no submission cap, data ownership | Formspree free tier |
| Tailwind `darkMode: 'class'` over `'media'` | Allows user override of OS preference | `'media'`-only (no toggle) |
| Defer cloud sync to v3 | Honors "no backend" v2 constraint; schema is sync-ready when we want it | Build sync now |
| Gallery reuses URL-fragment codec for hydration | DRY — one circuit-loading code path | Separate "load from gallery" path |
| Theme stored in localStorage (not cookie) | Static site, no server reads it | Cookie |
| Honeypot for spam (not reCAPTCHA) | Zero JS deps, accessible, sufficient for low-volume site | reCAPTCHA v3 |
