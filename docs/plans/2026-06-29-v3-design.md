# Quantum v3 — Design Doc

*Created: 2026-06-29 via brainstorming session.*
*Status: Design — not yet scoped into `.planning/ROADMAP.md` phases.*
*v2 context: v2 is code-complete (theme, feedback, gallery, polish).
v3 starts after v2 ships.*

---

## 1. Problem Statement

### In plain language

v1 earned trust with real, inspectable math. v2 closed the comfort
gaps (theme, save, feedback). The next gap every reader will hit is
the same one:

> *"OK I understand qubits, gates, and even Deutsch's algorithm — but
> where does this actually bite reality?"*

The current site teaches the building blocks beautifully but stops
short of the payoff. The Deutsch essay is a pedagogically perfect toy
with no real-world consequence. Without that payoff, the site risks
being read as "lovely intuition pump, but I still can't explain to a
PM why we should care."

v3 fills that gap by **pairing the canonical algorithm canon with the
practical use case each algorithm unlocks.** Every v3 essay teaches
the algorithm AND the use case in the same scroll, with widgets that
make both halves interactive.

### Who is affected

- **Working devs** who finished v1's seven essays and are asking
  *"so what?"* — they get the answer.
- **Skeptical engineers** debating whether quantum is worth keeping
  on their professional radar — they get a defensible "yes / no /
  not-yet" per industry.
- **Educators / advocates** pointing colleagues at v1 — they get a
  v3 link that ends the "but where does this matter?" follow-up
  conversation.

### What exists today and why it stops short

- v1 algorithm essay (`/deutsch`) teaches the *mechanism* but the
  use case is *"distinguish a constant function from a balanced one
  in one query"* — true, elegant, and zero industrial relevance.
- v1's "algorithm essays use the 'why should I care' opener" key
  decision is honored but the opener is a single paragraph, not a
  full interactive use-case half.
- The site has no story for the algorithms most-asked about by
  readers: Grover, Shor, QFT, teleportation, and the variational
  near-term workhorses (VQE / QAOA).

### What we are NOT doing in v3

(Detail in §10.) Most importantly: not building a "real quantum
hardware integration." The pedagogical bridge to real hardware is
the Qiskit-text export from the sandbox — one-way, no auth, no API.

## 2. Solution Overview

### In plain language

**v3 = 5 paired essays + foundation infra + 3 small deferred-item
wins, structured as 6 phases.**

Every essay is two halves under one scroll:

1. **Algorithm half** — intuition → widget → math, identical rhythm
   to v1 essays. Real in-browser simulator math driving real widgets.
2. **Use-case half** — "now you have the engine, here's where it
   bites reality": a use-case-specific widget + a defensible (cited,
   no marketing-deck claims) explanation of where the algorithm
   does or doesn't matter today.

The two halves are bound by a hand-off paragraph; the essay never
ends without the reader having played with both.

### The 5 essay pairs

| # | Algorithm | Use case |
|---|---|---|
| 1 | Teleportation         | Quantum networks / quantum internet |
| 2 | Superdense coding     | Bandwidth bounds (Holevo) — completes the communication arc |
| 3 | Grover                | Unstructured search + the √N reality check |
| 4 | Shor + QFT (merged)   | Post-quantum cryptography threat |
| 5 | VQE                   | Quantum chemistry / drug & materials discovery |

### Visual map

```
                     ┌─────────────────────────────────┐
                     │  v1 site + v2 (theme, gallery,  │
                     │  feedback) — unchanged          │
                     └──────────────┬──────────────────┘
                                    │
                  ┌─────────────────┼─────────────────┐
                  │                 │                 │
                  ▼                 ▼                 ▼
       ┌────────────────┐  ┌────────────────┐  ┌────────────────┐
       │ Foundation     │  │ 5 paired       │  │ Closing wins   │
       │ infra          │  │ essays         │  │                │
       │ (Phase 1)      │  │ (Phases 2-6)   │  │ (Phase 6 tail) │
       └────────────────┘  └────────────────┘  └────────────────┘
                │                  │                  │
                ▼                  ▼                  ▼
       Qiskit text export   Teleport / Superdense  Concept-map
       + bundle-size CI     Grover / Shor / VQE    progress indicator
                            (each = algo+use case)  + v3 launch
```

### Carries forward from v1 + v2 (no re-litigation)

- Static, no-backend. No new runtime deps beyond what each algorithm needs.
- Vanilla TS simulator (still 4-qubit cap — see §3.1 for why we
  *don't* extend it).
- Reactivity via in-repo `signal<T>`.
- Sandbox visualization re-use pattern (slim `MiniBloch`-style
  components; never import sandbox hydrators coupled to the
  singleton store).
- Build-time circuit encoding via existing codec for any "remix in
  sandbox" CTAs.
- Test mirrors for every canonical list change (concept-map nodes,
  nav graph, sandbox-link starters) touched in the same commit.
- **No analytics. Ever.** (Concept-map progress indicator is
  `localStorage`-only; PROJECT.md "no tracking" promise gets a
  one-line clarification to distinguish local-only progress from
  tracking.)

### Core v3 design principle

**Every use-case half must have at least one widget.** Otherwise the
second half becomes a written-essay-tacked-onto-an-interactive-one
and feels off-vibe with the rest of the site. Use case + widget
makes the case for "this matters" interactively, not by assertion.

## 3. Architecture & Components

### 3.1 Simulator scope — stays at 4 qubits

The brainstorming session opened with a 6-qubit extension penciled
in for Shor. We removed it once Qiskit-text export was on the table.

**Why 4 qubits is enough for v3:**

| Algorithm  | Qubits needed in-browser | Where the bigger version lives |
|---|---|---|
| Teleportation | 3 | — |
| Superdense    | 2 | — |
| Grover (N=4)  | 3 (oracle + diffusion) | — |
| QFT visualizer | 4 (16 frequency bins) | — |
| Period-finding demo | 3-4 (tiny mod-exp) | — |
| Full Shor on N=15 | 12 (4 result + 8 ancilla) | **Qiskit export → real backend** |
| VQE on H2 | 2-4 | Larger molecules → pre-baked `MoleculeGallery` |

The Shor essay teaches **QFT** and **period-finding** as separate
live widgets at 4 qubits, then composes them into the full Shor
circuit *statically* with a prominent "Export to Qiskit" CTA. This
is stronger pedagogy than a watered-down in-browser Shor: *"You've
learned the parts. Here's the assembled algorithm. Now run it for
real."*

Sim's constructor guard (REQ-13) is untouched.

### 3.2 Phase 1 — Foundation

Two infra adds, no essay content. Both touch every later phase.

**Qiskit text export from sandbox circuits.**

```
src/lib/qiskit/
  export.ts          ← gate-by-gate Circuit → Qiskit Python text
  export.test.ts     ← round-trip + golden-file tests per gate
src/components/sandbox/
  QiskitExportButton.client.ts   ← sandbox toolbar button + modal
```

Generated output is plain text:

```python
from qiskit import QuantumCircuit
qc = QuantumCircuit(2)
qc.h(0)
qc.cx(0, 1)
qc.measure_all()
```

Sandbox-supported gates today (H/X/Y/Z/S/T/Rx/Ry/Rz/CNOT) map 1:1
to Qiskit. Any new gate added during a v3 phase (e.g. Toffoli for
Grover's oracle) MUST be added to both the simulator AND the
exporter in the same commit — a `tests/quantum/qiskit-export.test.ts`
golden test for every supported gate makes this drift-proof.

**Bundle-size CI gate.**

```
scripts/check-bundle-size.mjs   ← parses Astro build output; fails CI on regress
.github/workflows/ci.yml        ← invokes the script after `astro build`
```

Per-route budget table lives in the script. Initial values mirror
v2's bundle audit: essay routes ≤ 6 KB initial JS (current ~4 KB +
50% headroom); `/sandbox` and `/gallery` get their own ceilings.
New v3 routes get budgets set at each phase's plan-phase.

### 3.3 Phase 2 — Teleportation + Quantum Networks (flagship)

```
src/pages/
  teleportation.astro
src/components/
  MultiBlochPanel/
    index.astro
    Panel.client.ts      ← 3 Bloch spheres side-by-side; mixed-state aware (§3.3a)
  ProtocolStepper.astro  ← step through Bell prep → BSM → classical → correction
  NetworkDiagram.astro   ← Alice/Bob/Repeater interactive use-case widget
```

**Pedagogical arc:** entanglement-based information transfer →
why no quantum channel is needed for the data itself → how
real-world quantum networks would compose this protocol with
quantum repeaters.

#### 3.3a `MultiBlochPanel` must handle mixed states honestly

Bloch sphere only honestly represents *pure single-qubit states*.
Mid-teleportation, the three qubits are entangled — each qubit's
reduced density matrix is mixed. We already have
`src/lib/quantum/reducedDensity.ts`; the panel uses it to render
mixed states as **vectors *inside* the sphere** (shorter arrow,
faded opacity, optional "mixed" badge), not on the surface.
Anything less is a pedagogical lie and would corrode v1's trust.

### 3.4 Phase 3 — Superdense + Bandwidth/Holevo

```
src/pages/
  superdense.astro
src/components/
  EncodingTable.astro    ← 2 classical bits → Pauli gate Alice applies → Bob's outcome
  HolevoBoundViz.astro   ← slider for "qubits sent" → max classical bits transmittable
                            (re-uses MultiBlochPanel + ProtocolStepper from Phase 2)
```

**Pedagogical arc:** the mirror of teleportation. Use entanglement
to send 2 classical bits with 1 qubit; then meet Holevo's bound and
realize entanglement doesn't break the strict 2-bits-per-qubit
classical limit on its own.

### 3.5 Phase 4 — Grover + Search Reality Check

```
src/pages/
  grover.astro
src/lib/quantum/
  grover.ts              ← oracle + diffusion helpers
src/components/
  AmplitudeBars.astro    ← signed-amplitude variant of ProbabilityBars (shows the oracle flip)
  GroverIterator.astro   ← iteration stepper (1..⌊π√N/4⌋) — amplitudes climb then collapse
  SearchComparison.astro ← slider for N → classical N/2 vs quantum √N side-by-side
```

**Pedagogical arc:** amplitude amplification → √N speedup →
honest reality check: this is *not* the exponential headline grabs
imply; database search isn't broken; password hashes need only
~256-bit security to absorb the worst case.

### 3.6 Phase 5 — Shor + QFT + Post-Quantum Crypto Threat (densest)

```
src/pages/
  shor.astro
src/lib/quantum/
  qft.ts                 ← QFT + inverse
src/components/
  QFTVisualizer/         ← state → frequency-domain bars; animated; user-driven input
    index.astro
    Viz.client.ts
  PeriodFinding.astro    ← classical vs quantum period detection on a small mod-exp
  FactoringStepper.astro ← STATIC narrated walkthrough of Shor-on-N=15 + "Export to Qiskit" CTA
  RSACountdown.astro     ← RSA-1024/2048/4096 → classical-years vs quantum-once-NISQ-scales;
                            "is your codebase quantum-safe?" with NIST PQC links
```

**Pedagogical arc:** Fourier transform of quantum amplitudes → it
extracts hidden periods → period-finding is the heart of factoring
→ post-quantum cryptography is a real ~10-year IT planning concern,
not science fiction. The bridge to "go run real Shor" is the
Qiskit export at the end of `FactoringStepper`.

**Owned framing (not apologized for):** the essay hook explicitly
sets the expectation — *"You'll build the engine in this essay;
then run the real thing in Qiskit on actual quantum hardware."*

### 3.7 Phase 6 — VQE + Quantum Chemistry + Launch

```
src/pages/
  vqe.astro
src/lib/quantum/
  vqe.ts                 ← tiny classical optimizer (plain gradient descent, ~50-100 LOC)
src/components/
  AnsatzCircuit.astro    ← parameterized H2 ansatz; θ sliders bound to Ry/Rz
  EnergyLandscape/       ← 2D heatmap of energy vs (θ₁, θ₂); drag optimizer ball OR auto-descend
    index.astro
    Landscape.client.ts
  ChemistryReadout.astro ← bond length, molecular energy, comparison to true ground state
  MoleculeGallery.astro  ← pre-baked H2 + LiH + small-molecule VQE results
```

**Pedagogical arc:** variational/hybrid quantum-classical loop →
energy minimization → why this is the most credible near-term
industrial use case (chemistry, materials, drug discovery) → and
what's *still* hard (only small molecules today; HF/CCSD reality
check).

**Closing-work bundled in Phase 6:**

```
src/lib/progress.ts                          ← visited-flag per essay slug, localStorage
src/components/ConceptMapProgress.client.ts  ← checkmarks on existing ConceptMap nodes
```

Plus: v3 Lighthouse audit (mobile a11y ≥ 95 on all 5 new routes,
both themes), v3 announcement draft (`LAUNCH-ANNOUNCEMENT.md`
mirroring v1+v2 pattern), bundle audit per Phase 1's CI gate.

### 3.8 Aggregate count

- **5 new pages** (`teleportation`, `superdense`, `grover`, `shor`, `vqe`)
- **16 new components** (`MultiBlochPanel`, `ProtocolStepper`,
  `NetworkDiagram`, `EncodingTable`, `HolevoBoundViz`, `AmplitudeBars`,
  `GroverIterator`, `SearchComparison`, `QFTVisualizer`, `PeriodFinding`,
  `FactoringStepper`, `RSACountdown`, `AnsatzCircuit`, `EnergyLandscape`,
  `ChemistryReadout`, `MoleculeGallery`)
- **2 client modules in `src/components/`** — `sandbox/QiskitExportButton.client.ts`
  (Phase 1) and `ConceptMapProgress.client.ts` (Phase 6)
- **5 new lib modules** — `qiskit/export.ts`, `quantum/grover.ts`,
  `quantum/qft.ts`, `quantum/vqe.ts`, `progress.ts`

Comparable in size to v2 (16 components + 4 lib modules added).

## 4. Data Flow

### 4.1 Algorithm-half widget interaction (same as v1)

```
User in /<algorithm-essay>
   │
   ├──► Interacts with widget (slider, click, drag)
   │
   ├──► Widget calls into src/lib/quantum/<algorithm>.ts
   │
   ├──► Underlying state vector mutates via the existing simulator
   │
   └──► All sibling widgets subscribed via signal<T> re-render in lockstep
```

Identical to v1 essays. No new pattern.

### 4.2 Use-case-half widget interaction

Use-case widgets (`NetworkDiagram`, `SearchComparison`,
`RSACountdown`, `MoleculeGallery`, `HolevoBoundViz`) are **mostly
self-contained** — they consume static data (pre-baked benchmarks,
NIST timelines, molecule ground-state energies) and respond to user
controls (sliders, dropdowns) with local component state.

They do NOT subscribe to the essay's quantum-state signal, because
the use case is a *consequence* of the algorithm, not a live
read-out of it. This boundary keeps each widget testable in
isolation.

### 4.3 Qiskit export

```
User in /sandbox (or any essay's CircuitView)
   │
   ├──► Clicks "Export to Qiskit" button
   │
   ├──► src/lib/qiskit/export.ts walks the Circuit gate list
   │
   ├──► Renders a Python text block in a modal: copy-to-clipboard button
   │
   └──► User pastes into Jupyter / IBM Quantum / qiskit.org sandbox
```

One-way, build-time-safe (no runtime calls). The exporter never
talks to a network.

### 4.4 Concept-map progress

```
User finishes scrolling an essay (>= 90% scroll depth, fires once)
   │
   ├──► src/lib/progress.ts sets localStorage["quantum/progress/<slug>"] = timestamp
   │
   └──► On homepage load, ConceptMap reads the progress map and renders
        a checkmark badge on each visited node
```

No network, no aggregate count beamed anywhere. Per-device only.
Clearing localStorage resets progress. PROJECT.md's "no analytics"
promise gets a one-line clarification: *local-only progress is OK;
tracking is not.*

## 6. Error Handling & Edge Cases

### Simulator boundary

| Case | Behavior |
|---|---|
| User configures a v3 essay's circuit to exceed 4 qubits (shouldn't be possible in v3 essays' embedded widgets, but possible if they get to sandbox via "remix") | Existing REQ-13 constructor guard already throws. Sandbox toolbar already disables "+ Qubit" at the cap. |
| QFT input with a non-power-of-2 dimension | Pad to nearest power of 2; render an inline note. |
| Period-finding on a function whose period exceeds the register | Show "no period found in this window — try a smaller `a`"; do not crash. |
| VQE optimizer fails to converge in N steps | Show the trajectory truncated, last energy, and "didn't converge — try a different initial θ." Never spin forever. |

### Qiskit export

| Case | Behavior |
|---|---|
| User exports a circuit with an unsupported gate | The translator throws a clear `UnsupportedGateError`; UI catches and shows "this circuit uses gate `Foo` which isn't in the Qiskit export yet — please file a bug." Honeypot-protected feedback CTA inline. |
| User exports an empty circuit | Render an empty-but-valid `QuantumCircuit(n)` text with a friendly comment. |
| Browser clipboard API unavailable | Fall back to selectable text + "select all" hint. |

### Use-case widgets

| Case | Behavior |
|---|---|
| `SearchComparison` slider pushed to N=2^53 | Clamp at 2^32; show "(beyond IEEE-754 safe-integer range)" footnote. |
| `RSACountdown` quantum-feasibility estimate | Render with explicit "as of NIST 2024 estimates"; never assert specific dates. |
| `MoleculeGallery` pre-baked energy missing for a selected molecule | Disable selector entry with tooltip "not yet computed"; do not 500. |
| `EnergyLandscape` evaluator slow on mobile | Debounce drag to 50 ms; on low-end devices, lower grid resolution to 25×25. |

### Progress indicator

| Case | Behavior |
|---|---|
| `localStorage` blocked | Silent no-op; progress badges just don't render. |
| User clears storage / new device | Progress resets. (By design — no cross-device sync; tracks v2's IndexedDB choice for the same reason.) |
| Concept-map node added for an essay the user already scrolled before v3 ships | First post-v3 visit fires the badge. Acceptable — old visits weren't recorded. |

## 7. Testing Strategy

### Per-algorithm correctness (non-negotiable, same bar as v1)

Every algorithm gets vitest coverage against textbook-known outputs.

```ts
// tests/quantum/teleportation.test.ts
test("teleporting |+⟩ through Bell pair lands |+⟩ on Bob after correction", () => { ... });

// tests/quantum/grover.test.ts
test("Grover on N=4, marked=2, 1 iteration → P(2) ≈ 1.0", () => { ... });

// tests/quantum/qft.test.ts
test("QFT of |0000⟩ = uniform superposition", () => { ... });
test("Inverse QFT undoes QFT", () => { ... });

// tests/quantum/vqe.test.ts
test("Gradient descent on H2 ansatz converges to ground state ± 1e-3", () => { ... });

// tests/qiskit/export.test.ts
test("H + CNOT exports to the canonical Qiskit Bell-pair text", () => { ... });
// One golden test per supported gate.
```

### Widget integration

Vitest + happy-dom for component-level tests. Verify slider changes
propagate to readouts; `MultiBlochPanel` correctly shrinks the
arrow when a qubit's reduced density matrix becomes mixed; static
widgets render the expected text.

### Mirror tests

Same pattern as v1+v2: every canonical list updated for v3 has a
mirror test asserting completeness.

- `tests/essays/concept-map.test.ts` — asserts 12 nodes (was 7)
- `tests/essays/nav-graph.test.ts` — asserts new prev/next chain
- `tests/essays/sandbox-links.test.ts` — adds starters for
  teleportation, superdense, Grover (not Shor — too big)
- `tests/qiskit/export.test.ts` — asserts every sandbox-supported
  gate has a Qiskit translation (drift-proofs gate additions)

### Pedagogy validation (same as v1)

The truest test: hand each v3 essay to a working dev who hasn't
studied quantum, ask them to (a) explain the algorithm back and (b)
explain *why someone outside academia should care.* If both clicks,
the essay works. If only (a) clicks, the use-case half needs
another pass.

### CI gates

- All vitest tests pass
- `npm run build` succeeds (16 → 21 pages)
- `scripts/check-bundle-size.mjs` passes (every route within budget)
- Lighthouse audit (manual, Phase 6) — mobile a11y ≥ 95 on every
  v3 route in both themes

## 8. Open Questions & Risks

### Open questions (resolve at plan-phase)

1. **Qiskit export — every circuit, or just Shor?** *Recommendation:*
   every sandbox circuit + every essay's embedded `CircuitView`.
   The translator is gate-by-gate; same code path. Strengthens v3's
   "now go do this for real" theme across all essays, not just Shor.
   Natural sibling to v1's "remix in sandbox" CTA. Decide at Phase 1
   plan-phase.
2. **v3 sandbox-link starters per essay** — Teleportation, Superdense,
   Grover circuits fit in the 4-qubit sandbox; should they get
   one-click "remix in sandbox" starters like v1 essays did? Lean:
   yes for those three; no for Shor (circuit too big). Decide at
   each phase's plan-phase.
3. **VQE optimizer algorithm choice** — plain gradient descent
   (simple, inspectable, visible) vs. SPSA (industry-realistic but
   more code). Lean: gradient descent for v3; SPSA → v4 stretch.
   Decide at Phase 6 plan-phase.
4. **PROJECT.md "no tracking" wording for progress indicator** —
   `localStorage["quantum/progress/<slug>"]` is per-device, no network,
   no PII. Update PROJECT.md's "no analytics. ever." entry with a
   one-line clarification that local-only progress doesn't count as
   tracking. Cheap doc edit; do at Phase 1.
5. **Concept-map layout after 12 essays** — current force-directed
   layout audit during Phase 6 polish. If cluttered, group nodes
   by track (Foundations / Algorithms / Applications). Visual
   refactor only.

### Risks (from reverse brainstorming)

| Risk | Severity | Mitigation |
|---|---|---|
| Shor's "no in-browser factoring" reads as cop-out | High | Essay hook owns it: *"You'll build the engine; then run the real thing in Qiskit."* No apology in copy. |
| `MultiBlochPanel` rendering entangled qubits as pure states (lie) | High | Use `reducedDensity.ts`; render mixed states as vectors *inside* the sphere with a "mixed" badge. |
| VQE classical-optimizer credibility | Medium | Vanilla TS, ~50-100 LOC, inspectable in DevTools — same trust bar as the simulator. |
| Use-case half pacing strain (essays bloat) | Medium | Word budget per essay enforced at plan-phase. If a use-case half balloons, split into a sibling "deep dive" page; do not let the essay sprawl. |
| Hype-prone use-case content (RSACountdown, MoleculeGallery) | Medium | Every claim cited (NIST PQC timeline, HF/CCSD benchmarks). No marketing-deck assertions. |
| Qiskit-export gate-coverage drift | Low | Golden test per supported gate; new gates must touch sim + exporter in the same commit. |
| Bundle creep across 5 new routes | Low | Phase 1 CI gate per route; v3 route budgets set at each plan-phase. |
| `EnergyLandscape` live evaluation cost on mobile | Low | Debounce drag to 50 ms; coarse grid (25×25) fallback below LCP-low device threshold. |
| Concept-map UX after 12 nodes | Low | Phase 6 audit; track grouping refactor if needed. |

## 9. Suggested Sequencing — 6 phases

Each phase independently shippable. Foundation-first; flagship-essay-
second to validate the format before scaling; densest essay (Shor)
gets its own phase; closing essay (VQE) bundled with launch work.

| Phase | Scope | Why this slot |
|---|---|---|
| **v3-01** | Foundation: Qiskit text export, bundle-size CI gate | Both touch infra every later phase consumes. Ship before any essay. |
| **v3-02** | **Teleportation + Networks (FLAGSHIP)** | Validates "algo + use case" format end-to-end before scaling. v1 lesson: ship one essay before the rest. Extends v1's entanglement story → lowest pedagogical risk for format validation. |
| **v3-03** | Superdense + Holevo | Completes the communication arc; reuses Phase 2's `MultiBlochPanel` + `ProtocolStepper`. Tightest phase. |

## 11. Decisions Log

| Decision | Rationale | Alternative considered |
|---|---|---|
| Interleaved essays (algorithm + use case per essay) | Deepens v1's "why should I care" opener into real content; one essay = one complete payoff | Algorithm-only essays (v3 wishlist as-was) |
| Curated 5 pairs (kept wishlist + added VQE) | Wishlist algorithms have varying use-case strength; VQE adds the strongest near-term industrial story (chemistry) | Original 5-wishlist as-is; or swap superdense for QAOA |
| Keep simulator at 4 qubits | Qiskit export handles the only algorithm that exceeds it (Shor); every other v3 essay fits | 8-qubit extension for simplified Shor |
| Merge QFT into Shor's essay | QFT is rarely a standalone story; teaching it as Shor's engine gives both a payoff | Separate QFT essay |
| Qiskit export as Shor's pedagogical bridge | Stronger framing than watered-down in-browser Shor: *"You've learned the parts; now run the real thing"* | Apologetic "we couldn't fit it in-browser" framing |
| 6 phases, essay-per-phase, foundation-first | v1 lesson: ship one essay end-to-end first. v3 essays are denser than v1's, so one-per-phase rhythm fits | Mirror v2's 4-phase batched silhouette |
| Flagship = Teleportation (Phase 2) | Extends v1's entanglement story most directly → lowest pedagogical risk for format validation | Grover (more famous) or Shor (biggest "wow") as flagship |
| Bundle launch polish with VQE (Phase 6) | VQE is the novel/risky phase; closing work bundles cleanly with it; mirrors v1 Phase 5 | Dedicated 7th launch phase |
| Defer 8-qubit simulator extension entirely | Qiskit bridge makes it unnecessary | Add it to Phase 1 "just in case" |
| Defer E2E + pick-a-path again | Both were deferred from v2; not blocking v3; their absence won't be felt | Absorb them into v3 |
| Absorb Qiskit export + bundle CI + progress indicator | All three reinforce v3's algorithm focus and are cheap; high leverage | Defer some to v4 |
| Vanilla TS for VQE classical optimizer | Same inspectability bar as the simulator; ~50-100 LOC | Pull in scipy.optimize-equivalent JS lib |
| Use case widgets are self-contained (no shared signal subscription) | Use case is a *consequence* of the algorithm, not a live read-out; keeps testability | Wire every widget into the essay's signal |
| Concept-map progress indicator is localStorage-only | Honors "no analytics" promise (with one-line clarification) | Skip the indicator entirely |
| `MultiBlochPanel` renders mixed states inside the sphere | Truthful visualization; sphere only honestly shows pure single-qubit states | Render every qubit as a unit-length arrow (would be a lie) |
| Every algorithm widget runs real sim math; only the *composed* Shor circuit is static | Trust bar from v1 unbroken; static rendering only where in-browser execution is genuinely infeasible | Pre-bake more widgets (would erode trust) |
